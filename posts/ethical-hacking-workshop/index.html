THE DIFFERENCE BETWEEN IP ADDRESS AND MAC ADDRESS

IP Address: Logical, identifies devices across networks.
MAC Address: Physical, identifies devices within networks.

THE OSI (OPEN SYSTEM INTERCOMMUNICATION) MODEL:

  1. Physical Layer: Transfers raw bits over physical mediums.
  2. Data Link Layer: Manages node-to-node data transmission reliability.
  3. Network Layer: Routes data packets between networks.
  4. Transport Layer: Ensures error-free, complete data transfer.
  5. Session Layer: Manages and controls communication sessions.
  6. Presentation Layer: Formats, encrypts, and translates application data.
  7. Application Layer: Enables user and application interactions.

Here are common exploits at each layer of the OSI model:

  1. Physical Layer:

    • Cable tapping (eavesdropping on wired communication).
    • Jamming (disruption of wireless signals).

  2. Data Link Layer:

    • MAC spoofing (impersonating another device’s MAC address).
    • ARP poisoning (manipulating ARP tables for interception).

  3. Network Layer:

    • IP spoofing (forging sender IP addresses).
    • Routing attacks (e.g., BGP hijacking).

  4. Transport Layer:

    • TCP SYN flood (denying services by exhausting resources).
    • UDP flood (overloading the system with UDP packets).

  5. Session Layer:

    • Session hijacking (taking over active sessions).
    • Man-in-the-middle (MITM) attacks targeting sessions.

  6. Presentation Layer:

    • Code injection (e.g., malicious scripts).
    • Cryptographic attacks (e.g., exploiting weak encryption).

  7. Application Layer:

    • SQL injection (manipulating database queries).
    • Cross-site scripting (XSS) and buffer overflows.

ARP (Address Resolution Protocol):

  • Purpose: ARP is used to map an IP address (logical address) to a MAC address (physical address) on a local network.
  • How it works:
    1. A device sends an ARP request asking, “Who has this IP?”
    2. The device with the matching IP responds with its MAC address.
    3. The ARP table stores this mapping for future use.
  • Security Issue: Vulnerable to ARP spoofing, where attackers trick devices into associating the wrong IP-MAC pairing, enabling eavesdropping or traffic redirection.

BGP (Border Gateway Protocol):

  • Purpose: BGP is a routing protocol that determines the best paths for data to travel between autonomous systems (AS), which are large networks typically controlled by ISPs.
  • How it works:
    1. Routers exchange routing information using BGP to advertise the networks they can reach.
    2. Based on policies and metrics, BGP decides the most efficient route for transmitting data.
  • Security Issue: Vulnerable to BGP hijacking, where malicious actors announce false routes, redirecting or disrupting traffic flow.

TCP VS UDP

Connection Type Connection-oriented (requires handshake) Connectionless (no handshake)
Reliability Reliable, ensures data delivery Unreliable, no guarantee of delivery

Subnetting: Dividing a network into smaller parts for manageable access to enhance performance and security

A subnet mask is a 32-bit number used in IPv4 networks to divide an IP address into two parts: the network address and the host address. The subnet mask helps routers and devices on the network know which part of an IP address refers to the network and which part refers to individual devices within that network. Structure:

The subnet mask consists of a series of 1s followed by 0s, where the 1s indicate the network portion and the 0s represent the host portion.
Example: 255.255.255.0

Common Subnet Masks:

/24 or 255.255.255.0: Common for small networks, where 8 bits are left for hosts.
/16 or 255.255.0.0: Common for medium-sized networks, leaving 16 bits for hosts.
/8 or 255.0.0.0: Common for larger networks, where 24 bits are left for hosts.

Example:

For the IP address 192.168.1.10 and subnet mask 255.255.255.0:

Network portion: 192.168.1.0
Host portion: .10 (device-specific part of the address)

Subnet masks help devices within the same network communicate directly, and those on different networks use routers to communicate.

DNS Resolving:

DNS Resolving refers to the process by which a domain name (like example.com) is translated into an IP address (like 93.184.216.34) that computers can understand to establish communication over a network.

Steps in DNS Resolving:

  1. DNS Query Initiation:

    • A user types a domain name (e.g., www.example.com) into a browser.
    • The browser sends a query to a DNS resolver (usually provided by the user’s ISP).

  2. Recursive DNS Query:

    • The DNS resolver receives the domain name and checks its cache to see if it already knows the IP address.
    • If not found, the resolver starts querying the DNS hierarchy starting from the root DNS servers.

  3. Query to Root Servers:

    • The DNS resolver asks a root DNS server for the IP address of the Top-Level Domain (TLD) server (e.g., for .com, the root server provides the IP for .com TLD servers).

  4. Query to TLD Servers:

    • The resolver queries the TLD servers for the authoritative nameserver that holds the information for the specific domain (example.com).

  5. Query to Authoritative DNS Servers:

    • The DNS resolver sends the query to the authoritative DNS server for the domain (e.g., example.com).
    • This server contains the domain’s actual A record (which provides the IP address for the domain).

  6. Returning the IP Address:

    • The authoritative DNS server responds with the IP address associated with the domain name (e.g., 93.184.216.34).
    • The resolver sends the IP address back to the client (browser).

  7. Caching:

    • The DNS resolver caches the IP address for future requests to speed up subsequent queries for the same domain.
    • The browser also caches the resolved IP address for a short time to avoid repeated DNS lookups.

Types of DNS Records:

  • A record: Maps a domain to an IPv4 address.
  • AAAA record: Maps a domain to an IPv6 address.
  • CNAME record: Provides an alias for a domain (e.g., www.example.com might point to example.com).
  • MX record: Specifies mail servers responsible for receiving emails for the domain.
  • NS record: Specifies the authoritative nameservers for a domain.
  • TXT record: Allows domain administrators to store arbitrary text for the domain (often used for validation and security).

DNS Resolver Types:

  • Recursive DNS Resolver: Queries for the domain resolution on behalf of the client, performing all steps as mentioned above.
  • Caching DNS Resolver: Caches DNS query results to improve lookup performance for frequently accessed domains.
  • Authoritative DNS Resolver: Provides answers for domains it is responsible for, containing records like A, MX, and CNAME.

DNS is vital in enabling users to access websites using human-readable names instead of remembering complex IP addresses.

HTTP Request Types (Methods):

HTTP request methods, often called HTTP verbs, define the desired action for the given resource. Here are the most common types:

  1. GET:

    • Purpose: Retrieves data from the server (e.g., a webpage, an image).
    • Characteristics: Safe (doesn’t modify data), idempotent (same response for repeated requests), and cacheable.

  2. POST:

    • Purpose: Submits data to the server to create or update a resource (e.g., submitting a form, logging in).
    • Characteristics: Non-idempotent (submitting the same request multiple times may produce different results), not cacheable.

  3. PUT:

    • Purpose: Updates a resource on the server (or creates it if it doesn’t exist).
    • Characteristics: Idempotent (repeated requests with the same data will result in the same outcome).

  4. DELETE:

    • Purpose: Deletes the specified resource from the server.
    • Characteristics: Idempotent (deleting the same resource multiple times produces the same result).

  5. PATCH:

    • Purpose: Partially updates a resource (unlike PUT, which replaces the entire resource).
    • Characteristics: Idempotent (if the same patch is applied multiple times, the result will be the same).

  6. HEAD:

    • Purpose: Similar to GET but only retrieves the headers (not the body).
    • Characteristics: Often used for checking if a resource exists or for fetching metadata.

  7. OPTIONS:

    • Purpose: Describes the communication options for the target resource (e.g., supported HTTP methods).
    • Characteristics: Used by clients to discover what methods a server supports without actually requesting the resource.

  8. CONNECT:

    • Purpose: Establishes a tunnel to the server, often used for SSL/TLS-encrypted connections.
    • Characteristics: Primarily used in proxy servers for setting up a connection for secure communication.

  9. TRACE:

    • Purpose: Performs a diagnostic trace of the path a request takes through the HTTP stack, returning the exact request as received by the server.
    • Characteristics: Used mainly for debugging and diagnostic purposes.

HTTP Response Status Codes:

HTTP response status codes are standardized codes returned by the server to indicate the outcome of the request.

  1. 1xx (Informational):

    • Example: 100 Continue
    • Purpose: Indicate that the request was received and the client should continue.

  2. 2xx (Successful):

    • Example: 200 OK, 201 Created, 204 No Content
    • Purpose: Indicate that the request was successfully processed by the server.

  3. 3xx (Redirection):

    • Example: 301 Moved Permanently, 302 Found, 304 Not Modified
    • Purpose: Indicate that the client must take additional action to complete the request (e.g., follow a different URL).

  4. 4xx (Client Error):

    • Example: 400 Bad Request, 404 Not Found, 403 Forbidden, 405 Method Not Allowed
    • Purpose: Indicate that the request had bad syntax or could not be fulfilled (e.g., the resource was not found or permissions were insufficient).

  5. 5xx (Server Error):

    • Example: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable
    • Purpose: Indicate that the server failed to fulfill the valid request due to an error on the server side.

HTTP Protocol:

HTTP (Hypertext Transfer Protocol) is an application layer protocol used by the web for transferring data between client and server. Key features of HTTP:

  • Stateless: Each HTTP request/response is independent and the server doesn’t retain any session information.
  • Text-based: HTTP headers and data are transmitted as text (although the data might be binary).
  • Protocol Version: The most common versions of HTTP are:
    • HTTP/1.0: Initial version, introduced basic features.
    • HTTP/1.1: Improved version with persistent connections, pipelining, and additional features.
    • HTTP/2: Faster, more efficient version with multiplexed streams and binary data transfer.
    • HTTP/3: A new version based on QUIC (Quick UDP Internet Connections) for faster performance, especially in mobile networks.

Key Features of HTTP:

  • Request-Response Model: Clients send requests, servers send back responses.
  • Headers: Both requests and responses include headers for metadata (e.g., content type, caching policies).
  • Body: The body holds the data being transmitted (like HTML content in responses, or form data in requests).

TLS HANDSHAKE:

  The **three-way handshake** is a fundamental process in establishing a reliable connection in TCP/IP communication. It involves three steps between a client and a server before data transfer can take place. Here's how it works:

  1. SYN (Synchronize) – The client sends a SYN (synchronize) message to the server to initiate a connection. This message contains a random initial sequence number (ISN) which will be used to establish the sequence of data to be transferred.

    Example:

    • Client → SYN: Client wants to establish a connection with the server.

  2. SYN-ACK (Synchronize-Acknowledge) – The server responds by sending a SYN-ACK message back to the client. This means the server is willing to accept the connection, and it acknowledges the client’s SYN request. The server also includes its own initial sequence number (ISN) and acknowledges the client’s ISN by incrementing it by 1.

    Example:

    • Server → SYN-ACK: Server confirms its intention to establish a connection and acknowledges the received SYN.

  3. ACK (Acknowledge) – The client sends an ACK (acknowledgment) message back to the server to confirm the server’s SYN-ACK message. This acknowledges the receipt of the server’s sequence number and confirms the connection.

    Example:

    • Client → ACK: Client acknowledges the server’s message.

Once this three-way handshake is complete, the connection is established and data can be transmitted.

To summarize:

  • Step 1: Client → SYN (Client initiates with a sequence number)
  • Step 2: Server → SYN-ACK (Server acknowledges and sends its own sequence number)
  • Step 3: Client → ACK (Client acknowledges the server)

This handshake ensures both sides agree on sequence numbers and synchronization, providing a reliable foundation for data communication.

Checking for Data Breaches: Have You Been Pwned?

One of the first steps in securing your digital identity is identifying if your information has been exposed in a data breach. Services like Have I Been Pwned? allow users to check if their email or other credentials have been compromised.

Additionally, there are Telegram bots designed to gather and notify users about breached data, offering another layer of vigilance to monitor online security. It’s crucial to remember, though, that while tools like these are helpful, using strong, unique passwords and enabling multi-factor authentication (MFA) are essential steps for online safety.

Practical Cybersecurity Tools: ZPhisher

The session introduced ZPhisher, a powerful open-source tool for creating phishing websites. With a range of templates for different services, ZPhisher demonstrates how attackers can craft fake sites to harvest login credentials.

Key Takeaways:

  • This tool highlights the importance of verifying URLs before entering sensitive information.
  • For professionals, it serves as a resource for understanding how attackers exploit human error, and by extension, how to mitigate phishing risks.

SQL Injection Vulnerabilities: A Case Study

During the session, a real-world example of SQL injection vulnerability was shown on CBHE Delhi’s Admin Page. SQL injection remains one of the most common web application vulnerabilities, allowing attackers to manipulate backend databases by injecting malicious SQL statements into input fields.

Steps Demonstrated:

  1. Testing the input fields for vulnerabilities.
  2. Exploiting these fields to access unauthorized data.

Prevention Measures:

  • Use prepared statements and parameterized queries.
  • Employ web application firewalls (WAF) to block malicious inputs.
  • Regularly audit your web applications for vulnerabilities.

Manipulating E-commerce Transactions with Burp Suite

The session also explored the dangers of Man-In-The-Middle (MITM) attacks using Burp Suite. By intercepting and modifying HTTP/HTTPS traffic between a client and an e-commerce website, attendees learned how attackers might tamper with transactions.

Insights Shared:

  • Attackers could adjust transaction amounts or manipulate purchase details.
  • Weak or poorly secured APIs often make these attacks possible.

Prevention Techniques:

  • Employ TLS (Transport Layer Security) with robust encryption.
  • Monitor traffic logs for anomalies that may indicate MITM activities.
